# Security

## Authentication

All service endpoints are protected by JWT bearer tokens. The JWT must include the following claims — missing any of them returns HTTP 401:

| Claim | Requirement |
|---|---|
| `sub` | Caller identity (string) |
| `exp` | Expiry time (Unix timestamp) |
| `iat` | Issued-at time (Unix timestamp) |
| `jti` | Unique token ID (UUID) — used for blacklisting |

Tokens are signed with `HS256` by default (`ADG_JWT_ALGORITHM`). Use RS256 for production deployments with a dedicated key pair.

### Token blacklisting

Revoked tokens are tracked in an in-process blacklist keyed by `jti`. The blacklist entry expires automatically when the token's `exp` passes (lazy TTL purge). Revoke a token:

```bash
curl -X POST http://controller:8080/auth/revoke \
  -H "Authorization: Bearer $ADMIN_JWT" \
  -d '{"jti": "the-jti-to-revoke", "exp": 1234567890}'
```

Note: the in-process blacklist does not survive service restarts. For persistent revocation, use short token TTLs (15 minutes) and a Redis-backed blacklist (planned).

## Authorization (RBAC)

JWT `roles` claim controls access:

| Service | Role | Permitted endpoints |
|---|---|---|
| Controller | `controller:read` | `/lures`, `/rules`, `/realism`, `/coverage`, `/enrich/ip`, `/techniques/*`, `/cmdb/assets` |
| Controller | `controller:write` | `/events` |
| Controller | `controller:admin` | `/rules/reload`, `/ad/*`, `/auth/revoke` |
| Token Engine | `token:read` | `/tokens`, `/bundle` |
| Token Engine | `token:write` | `/tokens`, `/access/{token_id}` |
| Token Engine | `token:admin` | `/rotate` |
| SOAR | `soar:write` | `/alerts`, `/playbooks` |

## Transport security

### mTLS

When `ADG_*_MTLS_REQUIRED=true` (default), services require the ingress to set `X-SSL-Client-Verify: SUCCESS`. Configure your ingress controller (nginx, Traefik, Envoy) to perform client certificate verification and set this header.

To temporarily disable during incident response:
```bash
ADG_MTLS_REQUIRED=false  # restart required
```

### Security response headers

Every response includes:

```
Strict-Transport-Security: max-age=63072000; includeSubDomains
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: no-referrer
Cache-Control: no-store
```

## Secrets management

### Vault (recommended)

```bash
# Store secrets
vault kv put secret/adg/controller \
  jwt_secret="..." \
  abuseipdb_key="..." \
  siem_token="..."

# Service reads on startup via ADG_VAULT_ADDR + ADG_VAULT_TOKEN + ADG_VAULT_SECRET_PATH
# Vault values take precedence over environment variables
```

### Kubernetes Secrets (alternative)

```bash
kubectl create secret generic adg-controller-secrets \
  --from-literal=ADG_JWT_SECRET=... \
  --from-literal=ADG_SIEM_TOKEN=...
```

Mount as env vars in the pod spec — never pass as plain `env` in manifest files committed to git.

## Sensitive data redaction

Before any alert is emitted to a SIEM, `redact_dict()` scans the entire payload recursively. It masks:

**By field name** (case-insensitive, 30+ names):
`password`, `passwd`, `pass`, `pwd`, `secret`, `secret_key`, `client_secret`, `token`, `access_token`, `refresh_token`, `id_token`, `bearer`, `api_key`, `apikey`, `api_secret`, `auth`, `authorization`, `x-api-key`, `private_key`, `private_key_id`, `ssh_key`, `pem`, `credential`, `credentials`, `ssn`, `social_security`, `credit_card`, `card_number`, `cvv`, `cvc`, `connection_string`, `db_password`, `database_password`, `aws_secret_access_key`, `aws_session_token`

**By value pattern** (regardless of field name):
- AWS access keys (`AKIA[A-Z0-9]{16,}`)
- PEM private key headers
- GitHub PATs (`ghp_...`)
- GitLab PATs (`glpat-...`)
- Slack tokens (`xox*`)
- JWTs (`eyJ...eyJ...sig`)
- Bearer tokens
- Credit card numbers

Redacted values are replaced with `[REDACTED]`.

## DNS injection prevention

All DNS operations in `dns_providers.py` pass names and values through `_sanitise()` — a regex allowlist that only permits `[a-zA-Z0-9_\-\.]+`. Any value not matching this pattern raises `ValueError` before touching the DNS provider.

## Network segmentation

The `honeypots` Kubernetes namespace has a default-deny egress network policy. Honeypots can only reach the controller webhook endpoint. This containment ensures that:

1. Attacker-controlled traffic cannot egress the honeypot namespace to reach internal systems
2. Compromised honeypot containers cannot be used as pivot points

## Audit trail

Every API action is logged to the action audit log with:
- `request_id` (UUID, generated by `AuditMiddleware` or propagated from `X-Request-ID` header)
- `action` (e.g., `deploy_lure`, `reload_rules`, `disable_ad_account`)
- `subject` (JWT `sub` of the caller)
- `resource` (affected entity)
- `outcome` (`success` or `failure` with error detail)
- `timestamp`

Ship audit logs to an immutable SIEM index. Do not grant write access to the audit log path from within the service pod.

## Supply chain

- All Python dependencies are pinned to exact versions in `requirements.txt`
- Run `scripts/generate_sbom.sh` to produce a Software Bill of Materials (`sbom.json`)
- Run `scripts/scan_vulns.sh` to scan images for known CVEs
- Sign images with `scripts/sign_images.sh` and enforce signature verification in your registry policy
- CI runs `ruff` (lint), `mypy` (type checks), and `pytest` (unit tests) on every commit
